exception also two types :-
---------------------------------
1.checked exceptions  and  2.unchecked exceptions 

checked exceptions :-
----------------------------
1.A method will throw a checked exception when it wants the calling method.Whichever method is calling that method, it wants that method to know that I am throwing an exception.
Do something about it.So if I want the calling method to try and handle this exception or do something about this exception or to know about this exception, then I would use something called a checked exception.  

2. any exceptions that doesnot extends runtime exception and that extends exception is called checked exceptions.
     ex:- 
	 class Exception extends Exception{}
     Class  B Extends Exception{}----------------this is called checked exception.
	 
	 class checkedException extends Exception{}
	 class checkedException1  extends checkedexception{}
    

2.unchecked exceptions(run time exceptions) :- 
--------------------------------------------------------

1. then the calling method does not need to know, does not need to do anything about that specific exception.then we will use unchecked exception.

2.any exceptions that extends runtime exception that exception is called unchecked exceptions.
ex:- class runtimeException extends Exception{}
     Class  B Extends RutntimeException{}----------------this is called unchecked exception.

	 class uncheckedException extends RutntimeException{}
	 class uncheckedException1  extends uncheckedException{}



Q) What is the difference between Error and Expection?
error :- error which is not handled by programmer for example StackOverFlowErrpr, OutofMemoryError.

Exception:- Exception is used when a programmer can handle the exception.


Q) what is difference between Checked Exception and Unchecked Exception?

Unchecked Exception:- Any class that are subclasses of RunTimeException or extends RunTimeExceptions then thoses classes are called Unchecked exceptions.
checked Exception :- Any class that are not subclasses of RunTimeException or extends Exceptions then thoses classes are called Unchecked exceptions.

best practices of exception handling
-----------------------------------------------
1.Never Hide Exceptions
2.do not use exception handling for flow control in a program.they have signifcant performance impact.
3.think about user.what does the user wants to see if tehre is an exception?
4.think about support developer.what does the support developer need to debug the exception?
5.cheked exception:-can the calling method do something about this exception then it handle the exception and do something good about it?

   unchekedException:-if calling method doesnot do anything then better to go with uncheked eception
   If you look at spring framework, it chooses to make most of the exceptions, most of the Jdbc exceptions as unchecked exceptions because there is nothing you would be able to do about a Jdbc exception.

If a Jdbc exception is thrown, then all that we can do is throw a sample message to the user saying,okay, some database exception happened and then you would need the support developer to go and look
at what the cause of the exception is.
